# 让 AI 真的"看懂"你的项目：xdev 新增 /ask 命令

大家好。

前段时间写过 xdev 怎么把 AI 编程助手从"执行命令的手"升级成"会自主编排的工程师"。今天讲一个更小、更日常的升级：**问答**。

你有没有遇到过这种场景？

- 要改 `login.ts`，问 AI："这个文件被哪些地方依赖？改签名会不会炸？"它打开文件、看了两眼、给你一段模糊的"可能影响登录流程"。
- 接手一个陌生项目，问 AI："这个支付模块是什么架构？为什么这样设计？"它从 README 里抄两段，再加一段"基于现代微服务理念"的废话。
- 合并代码前问 AI："最近一周哪些模块变动最大？有没有隐藏的耦合风险？"它告诉你"需要进一步分析"。

问题不在 AI 笨，在于它**没有正确看项目的眼睛**。

打开几个文件凑出来的答案，和真正读懂项目结构再回答，是两件完全不同的事。

xdev 这次新增了一个命令专门解决这件事：

```
/xdev:ask 这次改动会影响哪些模块？
```

---

## /ask 是什么

一句话：**专职回答"这个项目现在怎么样"的命令。不写代码、不跑测试、不做假设，只负责把事实讲清楚。**

它的定位很窄：

| 适合问 | 不适合问 |
|--------|---------|
| 项目结构、模块关系、调用链 | "帮我写一个 XXX 功能" |
| 某个文件/模块的当前实现 | "这个 bug 该怎么修" |
| 最近的变更影响范围 | "帮我重构一下" |
| 架构选型、设计意图 | "跑一下测试看看" |

想让 AI 动手写代码？去用 `/xdev:full-dev`、`/xdev:bugfix`、`/xdev:iterate`。

`/ask` 只做一件事：**把现状说清楚，说准确，说完整。** 想改动，换个命令。

---

## 为什么需要一个专门的"问答"命令

AI 代码助手默认的问答策略，大多是"打开几个相关文件、读一读、组织语言回答"。

这套策略在简单问题上够用——问"这个函数干嘛的"，打开函数定义读一下就行了。

但稍微复杂一点的问题就不够了：

**"改 `login.ts` 的 signature 会影响哪些调用方？"**

这个问题的正确答案需要：
1. 解析整个代码库的调用图
2. 找出所有直接和间接依赖 `login.ts` 的文件
3. 按模块分类，标注影响等级

"打开几个文件读一下"能解决吗？不能。AI 只能打开它觉得相关的文件，但"觉得相关"本身就是瞎猜。

**"这个支付模块为什么采用这种设计？"**

这个问题的正确答案需要跨越代码、文档、commit 历史、issue 讨论。单靠代码扫描给不出"为什么"。

**`/ask` 做的第一件事，就是换一双眼睛看项目。**

---

## 三层回答策略

`/ask` 按问题性质走三条路径：

### 第一层：知识图谱（问关系用）

问**调用链、依赖关系、模块交互、设计意图**这种需要跨文件推理的问题，`/ask` 会去读 **Graphify** 生成的项目知识图谱。

Graphify 是一个开源工具，把代码库（加上 README、架构文档等）解析成结构化图：节点是函数/类/模块，边是调用/继承/引用关系，再叠加文档里提取的语义信息。

简单说：**AST 扫描 + LLM 语义提取 + 图数据库。**

有了这张图，"改 `login.ts` 会影响哪些模块"就是一次图查询——秒级返回，还能列出影响链路。

### 第二层：源码扫描（问事实用）

问**具体文件内容、当前实现、最近改动**这种"看代码就能回答"的问题，`/ask` 直接用 `ripgrep` + Git，不动 Graphify。

为什么？这类问题根本不需要图，走图反而慢。工具要用在刀刃上。

### 第三层：诚实未知

查完图、查完代码，证据还是不够，`/ask` 就在输出里明确写一块 **Unknowns**——说清楚哪里没证据，建议怎么补。

**不编，不猜，不硬答。** 这是 `/ask` 和其它问答命令最大的区别。

---

## 比直接用 Graphify 好在哪

先说清楚：**`/ask` 不是 Graphify 的替代品。Graphify 是原材料，`/ask` 是把原材料变成答案的流程。**

直接用 Graphify 会碰到四个问题，`/ask` 把它们都内化了：

### 问题一：新鲜度靠你自己判断

图是昨天建的。今天你合并了 50 个 commit。图还能用吗？

直接用 Graphify 的时候，你得自己记得跑 `graphify check-update`、看有没有文件变动、决定要不要 `graphify update`。忘了就拿到过时的答案。

`/ask` 把这件事自动化：每次回答前先检查**三个信号**——

```
graphify check-update  (Graphify 自己的变更检测)
        +
Git HEAD commit 比对  (版本是否前进)
        +
源文件 mtime 扫描     (有没有未提交改动)
```

任何一个信号触发，都会自动刷新图再回答。

之前有人建议用"30 分钟新鲜度缓存"，被我们拒了——**时间 TTL 毫无意义。** 你 1 小时没动代码，图就是最新的；你 5 分钟合并 50 commit，图马上过时。按文件状态判断才是对的。

### 问题二：成本不透明

Graphify 首次建图要跑 LLM 提取语义，可能几百万 token、十几分钟。语义刷新也一样。

直接用 Graphify 的时候，你不知道下一次操作要花多久、花多少钱。

`/ask` 的做法：**任何会产生成本的操作，前面报数。**

```
检测到 docs/ 有 12 个文件变动，需要语义刷新
  预估：~180k tokens，约 2 分钟
  继续 / 跳过？
```

首次建图同理。**不偷跑，不意外，不烧钱。**

### 问题三：降级路径靠你自己写

Graphify 没装？图建失败？服务挂了？

直接用 Graphify 的时候，这些情况得你自己判断要不要走备选方案。

`/ask` 内置降级：Graphify 不可用就自动回落到 `ripgrep` + Git，回答深度会打折（明确告诉你打折了），但一定给答案。

### 问题四："问 AI 要不要用 Graphify"是个伪问题

之前我们的问答命令有一步"是否使用 Graphify？"——用户反馈："既然装了 Graphify 就是为了用，问这一步有什么意义？"

对。**装 Graphify 本身就是授权。** `/ask` 现在默认用，唯一的退出方式是**意图守卫**：你在对话里说一次"这次不要用 Graphify"，后续整轮都跳过，不再反复问。

### 一张表对比

| 维度 | 直接用 Graphify | /ask |
|------|----------------|------|
| 新鲜度判断 | 手动 | 三重信号自动 |
| 成本披露 | 无 | 语义刷新/首次建图前明示 |
| 降级路径 | 无 | 自动回落到 rg + Git |
| 查询语法 | 需要学 | 自然语言 |
| 授权模式 | 每次问 | 装了即授权，意图守卫单次退出 |
| 回答格式 | 自由文本 | TL;DR / 证据 / 未知 / 下一步 |

**Graphify 是利器。`/ask` 是把利器自动化、规范化、透明化的使用姿势。**

## 输出格式：强制 Evidence 和 Unknowns

大部分 AI 问答的输出是一大段连贯的自然语言，看起来很顺，但你没法验证。

`/ask` 强制结构化输出：

```
## TL;DR
(1-2 行直接回答)

## Answer / Findings
(详细分析)

## Evidence
- file:line / graphify node / git commit
- file:line / graphify node / git commit
...

## Unknowns
- 哪块证据不足
- 建议补充什么

## Next steps
- 如果想深入，建议跑什么命令
```

**每一条结论都要能追溯到具体文件、具体节点、具体 commit。** 没有证据的部分写在 Unknowns 里，不混在 Answer 里。

这个格式看起来有点啰嗦，但它消灭了一类问题：AI 说得很有把握，你拿去用，发现根本没这回事。

## 四个典型应用场景

**场景一：改动影响分析（改代码前）**

```
/xdev:ask 把 login.ts 的 authenticate() 签名从 (user, pwd) 改成 (credentials) 会影响哪些地方？
```

`/ask` 走图查询：先列出所有调用方、再按模块分类、再标注直接/间接影响。10 秒拿到一张影响清单，比你手动 grep 调用点省 20 分钟。

**场景二：陌生项目快速上手**

```
/xdev:ask 这个项目的支付流程是怎么跑的？从下单到回调，涉及哪些模块？
```

对新接手的项目特别有用。`/ask` 会沿着调用链走一遍、引用实际代码位置、顺便把文档和 commit message 里的设计意图带出来。

**场景三：合并前风险扫描**

```
/xdev:ask 最近一周哪些模块被动过？有没有看起来相关但没一起改的文件？
```

`/ask` 会结合 Git 历史 + 图结构，找出**改动聚簇**和**孤立改动**。比代码审查多一个维度。

**场景四：技术债盘点**

```
/xdev:ask 项目里还有哪些模块在用老的 legacy API？哪些迁移完了？
```

跨文件语义问题的典型场景。`/ask` 走图 + 代码扫描双路，把迁移状态给你列清楚。

## 执行边界：为什么 /ask 不跑测试

`/ask` 允许的操作：
- 读文件、读 Git log
- 查 Graphify、刷新 Graphify 图

`/ask` **禁止**的操作：
- 改源码
- 跑测试、跑构建
- 改配置、改依赖

为什么这么严？

**回答问题不应该有副作用。** 你问"这个模块怎么实现的"，结果 AI 顺手跑了 `npm install` 把你的 `node_modules` 刷新了——这不对。

想让 AI 动手？用别的命令。`/ask` 只负责**看**，不负责**动**。

刷新 Graphify 图算副作用吗？算——但它是**回答当前问题所必需的**副作用，而且只写到 `graphify-out/` 这个专属目录，不碰你的源码。这个边界是经过仔细权衡的。

## 安装（30 秒）

xdev 的安装流程这次也一起简化了：

```bash
git clone --depth 1 https://github.com/Minokun/xdev.git ~/.xdev
~/.xdev/bin/install.sh claude      # Claude Code
~/.xdev/bin/install.sh windsurf    # Windsurf
~/.xdev/bin/install.sh all         # 双平台都装
```

`install.sh` 是幂等的——重跑会更新软链，已有的真文件不会被覆盖。想装到项目级别加 `--project`，自定义路径加 `--target`。

Graphify 是可选项，不装也能用 `/ask`（自动降级到源码扫描）。想体验完整能力：

```bash
pipx install graphifyy
# 或
uv tool install graphifyy
```

然后在项目根目录跑一次 `/ask` 任意问题，`/ask` 会自动检测到 Graphify 已装、提示首次建图成本、等你确认。之后每次回答前自动保持新鲜。

## 最后

`/ask` 不是什么革命性的新东西。它做的事情很朴素：**把"问 AI 项目现状"这件事，从"碰运气"变成"可信赖"。**

朴素的事情往往是最值钱的——你每天要问十几次的事情，如果每次都能快 5 分钟、准一点、少一次被误导，一个月就是一大笔时间。
