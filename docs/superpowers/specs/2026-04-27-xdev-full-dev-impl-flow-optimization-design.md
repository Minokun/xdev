# xdev full-dev-impl Flow Optimization Design

Date: 2026-04-27
Status: Draft for review (rev 4 — internal re-review fixes)

## Context

A recent `/xdev:full-dev-impl` run for the stock-analysis project took about 92 minutes before the main session stopped at a Task 3 subagent handoff. The log showed the slow path was not caused by a single long-running test or environment setup. The time came from orchestration overhead:

- task-level implementation agents for small units,
- repeated spec and quality review loops,
- review findings triggering new tests, fixes, commits, and re-reviews,
- a Task 3 subagent re-entering planning workflows instead of acting as a narrow executor,
- path drift such as `backend/backend/...`,
- main-session visibility gaps while subagents continued writing their own logs.

The user wants to keep quality as the default. This design keeps the quality bar, but moves checks to the right level: risk-gated, batch-aware, and bounded.

## Goals

1. Preserve quality-first behavior for high-risk code.
2. Reduce unnecessary review loops for low and medium-risk tasks.
3. Prevent implementation subagents from re-planning or creating duplicate planning files.
4. Prevent path mistakes before dispatching subagents.
5. Improve main-session visibility into subagent progress.
6. Keep the final quality gate intact: tests, health, drift check, and release checks still run.
7. Make the optimization measurable across runs (stage 4 wall time, review-loop count) so subsequent revisions are data-driven.

## Non-goals

- Do not remove TDD.
- Do not skip reviews for high-risk work.
- Do not make `/xdev:full-dev-impl` a fast-only workflow.
- Do not rely on reviewer output without verification.
- Do not redesign the full `/xdev:full-dev` design-stage workflow here.

## Recommended model

Use a **Quality-First Risk-Gated Pipeline**.

Default profile remains quality-first. Internally, each implementation task is classified by risk, then dispatched with a matching execution and review strategy.

Optional future profiles can expose the same machinery explicitly:

```text
/xdev:full-dev-impl --strict
/xdev:full-dev-impl --quality    # default
/xdev:full-dev-impl --fast
```

This spec focuses on the default quality profile.

## Risk levels

Risk is assigned per task and **persisted in the stage 3 plan file** as a required field on every task entry. This is required so that:

- session recovery can resume without reclassifying,
- the cross-tool handoff (`full-dev-design` in one model, `full-dev-impl` in another) preserves the classification,
- standalone `/xdev:full-dev-impl` runs read risk directly from the plan rather than guessing.

| Level | Name | Signals | Default execution |
|---|---|---|---|
| L0 | Trivial/local | docs, comments, tiny styling/config, no public behavior change | Main thread or narrow subagent, no task-level review |
| L1 | Normal feature logic | one module, clear tests, no shared contract, no persistence | TDD subagent, batch review |
| L2 | Shared contract | API route behavior, shared helper, cache contract, serialization, cross-module utility | TDD subagent, task spec review, batch quality review |
| L3 | High risk | finance/math, training data, security, auth, permissions, persistence, migrations, concurrency, deploy infra | TDD subagent, task spec + quality review, independent audit (see below) |

Classification is conservative. If uncertain between two levels, choose the higher level.

**Default when missing (legacy plans only)**: a **fresh** plan generated by the updated stage 3 must include `risk` on every task; missing fields are a hard plan-validation error and block stage 4. Only **legacy** plans (authored before this design) are allowed to omit the field, in which case `/xdev:full-dev-impl` defaults each missing entry to **L2** and records the inference in the stage 4 state file (`docs/state/full-dev--<branch>--<slug>.md`) under a `risk_inferred: []` array.

L2 (rather than L3) is the legacy default because L3 forces per-task review + audit sidecar; defaulting all legacy entries to L3 would block migration. The `risk_inferred` array makes the choice visible so reviewers can manually upgrade specific tasks.

**L3 independent audit** (referenced in the table) is defined as: after task review passes, dispatch a separate review subagent with a different prompt template focused on the L3 signal that triggered the classification (e.g., for finance/math, a numerical-correctness audit; for auth, a permission-boundary audit). The audit is read-only and produces a sidecar at **`docs/state/audits/<slug>/audit-task-NNN.md`** (where `<slug>` matches the stage 4 state file slug). It is mandatory for L3, not optional.

Audit sidecars are gitignored alongside other `docs/state/` artifacts. The ship workflow must explicitly remove the directory at successful release: `rm -rf docs/state/audits/${_SLUG}` immediately after `rm -f "${_STATE_FILE}"`. Do not rely on a single recursive glob; the state file lives in `docs/state/` directly while the audit dir is one level deeper.

**Orthogonality with Graphify**: risk classification drives **review depth**; Graphify project context resolution (see `full-dev.md` Graphify lifecycle) drives **understanding depth**. They are independent. A task packet **may** include Graphify query output as read-only context for the executor, but Graphify availability does not change risk classification or review triggers.

## State file format additions

All new structured data introduced by this design (`tasks_in_flight`, `false_positives`, `risk_inferred`) lives in **one fenced YAML block** appended to the existing markdown state file at `docs/state/full-dev--<branch>--<slug>.md`. The pre-existing markdown bullet section (`## xdev 会话状态`) is preserved unchanged for human reading and the existing recovery checks; the YAML block is for machine read/write only.

Layout:

````markdown
## xdev 会话状态
- **功能：** <slug>
- **工作流：** full-dev
... (existing bullets)

## stage 4 data

```yaml
tasks_in_flight: []
false_positives: []
risk_inferred: []
```
````

Write cadence: the YAML block is rewritten **only on phase transitions** (subagent enters or leaves a phase, or `status` changes). Pure `last_event_at` timestamp ticks stay in main-thread memory and are flushed to disk at most once every 30 seconds, to avoid I/O thrash and write contention when multiple subagents emit events concurrently.

Atomic write: use the same temp-file-then-rename pattern the workflow already uses for the existing state file, so a crash mid-update never leaves a half-written YAML block.

## Review policy

Review is no longer a fixed per-task double loop. It is triggered by risk.

| Level | Spec review | Quality review | Drift/Gatekeeper |
|---|---|---|---|
| L0 | none | none | final only |
| L1 | batch summary review. Sample 1 task per touched module (no total cap). If batch touches > 6 modules, split the batch first. See sampling rule below. | batch once | final only |
| L2 | once after task | once after batch | batch + final |
| L3 | after task | after task | task + final |

### L1 sampling rule

For L1 batches, the main thread selects samples deterministically:

1. Group completed L1 tasks by touched top-level module or route directory.
2. **Every touched module gets exactly one sample** (the task with the largest diff in that module). There is **no total cap** — module coverage is the invariant.
3. **If a batch touches more than 6 modules**, split the batch into smaller batches (each touching ≤ 6 modules) before sampling. This keeps review work bounded without skipping any module.
4. If any sampled task has a blocking HIGH finding, review the rest of the L1 batch in that module before proceeding to other modules.
5. If all sampled tasks pass, defer non-sampled L1 tasks in their respective modules to final Gatekeeper and health.

Example: 6 modules, 1 L1 task each → sample all 6, one per module. 9 modules → split into two batches (e.g., 5 + 4) and sample each batch independently. This replaces an earlier formulation that capped samples at 3, which contradicted the per-module-coverage invariant.

**Empty case**: if the batch contains no L1 tasks, skip this step entirely (no modules, no samples).

### Finding handling

| Finding | Action |
|---|---|
| CRITICAL/HIGH | Must fix before proceeding |
| MEDIUM correctness/data-loss/security/performance | Must fix before proceeding |
| MEDIUM maintainability/naming/local structure | Defer to batch review unless it blocks clarity |
| LOW | Record, do not block |
| Reviewer claim conflicts with code evidence | Verify in code; if disproven, record as false positive (see format below) and do not enter fix loop |

#### False-positive record format

The **stage 4 state file is the single source of truth** for false positives. Batch summaries and review reports may **reference** entries (by `task` + `finding_id`) but must not duplicate the record. This avoids drift between the two locations.

State file path: `docs/state/full-dev--<branch>--<slug>.md` (the same file the workflow already creates at stage 3; see `full-dev.md` session-state section).

False positives are recorded under a top-level `false_positives: []` array. Each entry has the schema:

```yaml
- task: task-NNN
  finding_id: <reviewer-assigned id or short slug>
  severity: HIGH | MEDIUM | LOW
  reviewer_claim: <one-line summary of the claim>
  evidence:
    path: <absolute path>
    lines: <start>-<end>
    note: <why the claim is wrong>
  recorded_at: <ISO timestamp>
```

Batch summaries cite entries as: `False positive: task-NNN/<finding_id> (see state file)` — nothing more.

### Review loop bounds

For a single task:

1. First review pass runs according to the risk policy.
2. If blocking findings exist, fix them with failing tests where appropriate.
3. Re-review once.
4. If blocking HIGH/CRITICAL remains after the second review, pause and escalate.
5. MEDIUM/LOW that remain after the second review are moved to batch review unless they affect correctness or data safety.

This prevents endless review-fix-review cycles while preserving the stop-the-line behavior for real correctness problems.

## Subagent role split

Implementation subagents become narrow executors.

They must not:

- invoke planning skills,
- create `task_plan.md`, `progress.md`, or `findings.md`,
- rediscover the whole project,
- edit files outside the task allowlist,
- run broad tests before targeted tests pass,
- commit unrelated generated files.

They must:

- read only the assigned task packet,
- verify exact paths before editing,
- write the failing test first unless the task is explicitly non-testable,
- run the exact targeted verification command,
- run the assigned related regression command,
- return one of `DONE`, `BLOCKED`, `NEEDS_REVIEW`, or `NEEDS_RECLASSIFY` with evidence.

### Risk-escalation return code

If during implementation the subagent discovers the task actually touches a higher-risk surface (e.g., a task tagged L1 turns out to cross an auth boundary, hit persistence, or change a shared contract), it must stop work and return:

```text
status: NEEDS_RECLASSIFY
proposed_risk: L2 | L3
reason: <one-line justification + file:line evidence>
```

The subagent must not silently upgrade scope or proceed with the original packet. The main thread re-classifies, updates the plan, re-routes the task per the new risk (serial dispatch + audit if L3), and reissues the packet.

### Task packet format

The main thread sends each implementation subagent a compact packet:

```text
Repository root: <absolute path>
Working directory: <absolute path>
Risk level: L1/L2/L3
Task id: task-NNN
Allowed files:
- <absolute path>
Tests allowed:
- <absolute path>
Reserved test functions: (omit this entire block when no shared test-file contract applies)
- <test_function_name>
Targeted command:
- <command>
Related regression command:
- <command>
Graphify context (optional, read-only):
- <path to focused graphify query output, omit if none>
Do not invoke planning skills.
Do not create planning files.
Do not edit outside allowed files.
Return status and evidence only.
```

The `Graphify context` line is included only when the main thread already has a focused `graphify query` result for this task; it is never built ad-hoc inside the executor. See orthogonality note in the Risk levels section.

## Path preflight

Before starting any implementation subagent, the main thread runs a path preflight.

Required checks:

1. Repository root exists.
2. Working directory exists.
3. Every planned modify-file exists.
4. Every planned new-file parent directory exists.
5. Test command cwd is explicit.
6. **CWD/path collision rule**: when the subagent's cwd is `<repo>/<X>`, no path in the task packet may begin with `<X>/`. This precise rule replaces the earlier heuristic about generic repeated segments, which would have falsely flagged legitimate names such as `apps/apps-foo` or `python/python-stub`. It exactly captures the observed `backend/backend` failure mode without false positives.
7. All paths in the packet are absolute or are explicitly relative to the declared cwd; no implicit `./`-relative paths.

If preflight fails, do not dispatch the subagent. Fix the task packet first.

## Dispatch strategy

The current rule says tasks <= 3 or all dependent tasks become serial. That is too blunt.

Use file and contract conflicts instead, with an explicit small-batch fast path so we don't reintroduce orchestration tax for trivial batches.

### Small-batch fast path

The gate must be measurable from the plan **before** dispatch. Use task-packet shape, not estimated diff size:

When a batch has **all** of:

- ≤ 2 tasks total,
- each task's `Allowed files` list ≤ 1 entry,
- each task's `Tests allowed` list ≤ 1 entry,
- no L3 task present,

the main thread executes the tasks serially in-process and skips subagent dispatch entirely. This preserves the speed the previous `tasks <= 3` shortcut gave us, without its blanket coverage. The earlier wording "planned diff < 50 lines" was unmeasurable before implementation and has been replaced.

### Conflict matrix (for batches outside the fast path)

| Condition | Strategy |
|---|---|
| Same exact file edit | serial |
| Same test file, different tests | main thread creates a shared test-file contract first, then implementations may run after the contract is stable |
| Shared new helper required by others | helper first, consumers after |
| Distinct files and stable contract | parallel |
| L3 high-risk task | serial by default unless clearly isolated |
| Batch fails after merge | identify conflicting task, revert only that task commit, re-run serially |

### Shared test-file contract

When multiple tasks need the same test file, the main thread owns a short contract phase before implementation agents start:

1. Create or update the test file skeleton with imports, fixtures, helper fakes, and named empty test sections.
2. Assign each task explicit test function names.
3. Record the contract in each task packet under `Reserved test functions`.
4. The contract is stable when the file compiles (or the test runner reports collected-but-skipped) and each reserved test name is unique.
5. **Commit the scaffolding before parallel dispatch** as `chore(task-NNN-test-contract): scaffold <test_file>`. Subagents must observe a clean working tree with the scaffold already in HEAD; otherwise they cannot see their reserved test names.
6. Implementation agents may fill only their reserved test functions and their allowed production files. They must not add, rename, or reorder other reserved test functions.

In the stock-analysis cache example:

1. Shared cache helper is L2 and must be stabilized first.
2. Route consumers can run in parallel once helper contract is stable, unless they edit the same route or test file.
3. Consumer tasks should not each run a full planning workflow.

## Main-session visibility

Main-session visibility has two layers: a **platform-neutral interface** (what the workflow guarantees) and **platform-specific implementations** (Claude Code, Windsurf).

### Platform-neutral interface

The main thread must, for every subagent it has dispatched:

1. Track `dispatched_at`, `last_event_at`, and inferred `phase` for each task.
2. Emit a progress heartbeat to the main session output **and** persist `tasks_in_flight` into the stage 4 state file's YAML block (see State file format additions above) at **`docs/state/full-dev--<branch>--<slug>.md`**. Each entry: `{ task_id, risk, dispatched_at, last_event_at, phase, status }`. Persistence cadence follows the rule defined in the State file format additions section (phase transition or 30 s timer; never on every event).
3. Trigger heartbeat when the task exceeds the **risk-aware threshold**: L1 = 5 min, L2 = 8 min, L3 = 15 min. Long-running thresholds before declaring `possibly stuck`: L1 = 10 min, L2 = 15 min, L3 = 25 min, all measured from `last_event_at`.
4. If `last_event_at` is still advancing within the threshold, report `active`. State file persistence ensures session recovery does not lose subagent progress.

### Possibly-stuck action

When `possibly stuck` triggers, reporting alone is not enough. The main thread must take action:

1. Attempt one automated recovery: kill the subagent and re-dispatch the same packet once.
2. If the second attempt also reaches `possibly stuck`, escalate to 🔴: stop the workflow, summarize the last `tasks_in_flight` snapshot, and ask the user whether to retry, abandon, or downgrade to main-thread execution.
3. Never silently leave a stuck subagent running while continuing other tasks; always quarantine first.

Heartbeat output format:

```text
Task task-NNN (Lk) still running: last event at HH:MM, current phase: <test|implementation|verification|review|unknown>.
```

Phase is inferred from the most recent subagent event:

| Latest event | Phase |
|---|---|
| writing or running expected failing test | `test` |
| editing production files | `implementation` |
| running targeted or related tests | `verification` |
| invoking review agent or processing review output | `review` |
| no recognized event | `unknown` |

### Claude Code implementation

Subagent log lookup is deterministic:

1. Start from the current session directory under `~/.claude/projects/<project-session>/`.
2. Check `subagents/*.jsonl` under that session.
3. If the main session is in a worktree, also check the matching worktree project directory under `~/.claude/projects/*--worktrees-<worktree-name>/<session-id>/subagents/*.jsonl`.
4. Match by the task title, task id, or first 120 characters of the subagent prompt.
5. Use the matched log's mtime as `last_event_at`.

### Windsurf implementation

Windsurf does not expose subagent jsonl logs. Use one of the following, in order of preference:

1. **Subagent-emitted phase markers**: each subagent prompt includes an instruction to echo `[xdev-phase] <phase>` whenever it transitions; the main thread reads these from subagent stdout.
2. **Tool-call cadence**: when no phase marker is available, treat any tool-call result returned to the main thread as a `last_event_at` update.
3. **Final-result fallback**: if neither is available, fall back to a single timeout per risk level and treat the task as `unknown` until completion.

In all platforms: a still-updating subagent is `active`, not `blocked`.

## Quality gates retained

Even with risk gating, these gates remain:

1. Targeted tests for each task.
2. Related regression tests for touched modules.
3. Batch-level review for L1+ work.
4. Gatekeeper drift check for L2/L3 batches and final implementation.
5. `health` in stage 5+6.
6. UI `qa` when UI files/routes are touched.
7. `ship` still performs pre-landing checks.

## Failure behavior

### Subagent path error

A failure counts as a setup/path failure when all conditions hold:

1. The subagent has not yet made any successful file edit.
2. The error text contains one of: `No such file or directory`, `File does not exist`, `String to replace not found` caused by wrong path context, `cd: no such file or directory`, or — exactly — a path that violates the cwd/path collision rule from path preflight (e.g., cwd `backend/` plus packet path `backend/...`).

The earlier 2-minute / 10-tool-call window has been removed: it false-flagged subagents that did legitimate environment setup (lockfile checks, status, dependency install) before their first edit. Condition 1 plus the targeted error strings are sufficient to identify path/setup failures without false positives.

When this happens:

1. Stop treating it as implementation failure.
2. Correct the task packet and path preflight.
3. Relaunch once.
4. If repeated, run the task in the main thread or a fresh narrow subagent.

### Review false positives

If a reviewer finding is disproven:

1. Record the entry in the stage 4 state file `false_positives: []` array using the schema above.
2. Reference the entry from the batch summary (`task-NNN/<finding_id>`); do not duplicate the body.
3. Do not modify code for that finding.
4. Do not re-review solely because of the false positive.

### Repeated review findings

If the same area receives two rounds of blocking findings:

1. Pause the task.
2. Summarize the failing invariant.
3. Decide whether the plan is wrong, the code boundary is wrong, or the task is too large.
4. Split or escalate instead of continuing local edits.

## Expected effect

For the observed stock-analysis run:

- Task 1 would still receive review because exception behavior in middleware affects API correctness.
- Task 2 would remain L2 and reviewed, but the review loop would be bounded.
- Task 3 would skip planning-with-files inside the subagent and avoid `backend/backend` path drift through preflight.
- Route consumer tasks after the shared cache helper could run in parallel when file conflicts allow it.

Estimated runtime improvement for similar work: about 90 minutes down to 45-60 minutes while preserving quality gates for shared cache/API/finance-sensitive behavior.
